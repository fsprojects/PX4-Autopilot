### `PX4.Math.signNoZero`

| Lean name | C++ name | C++ location | Correspondence | Notes |
|-----------|----------|--------------|---------------|-------|
| `PX4.Math.signNoZero` | `math::signNoZero<T>` | [`src/lib/mathlib/math/Functions.hpp#L52`](../src/lib/mathlib/math/Functions.hpp#L52) | **abstraction** | Integer model; C++ uses bool subtraction |

**Divergences**:
- **Type**: C++ returns `int` from `bool` subtraction; Lean returns `Int`.
- **C++ expression**: `(T(0) <= val) - (val < T(0))` computes as `1 - 0 = 1` or
  `0 - 1 = -1`. The Lean model `if 0 ≤ val then 1 else -1` captures the same logic.
- **Float NaN**: `0 ≤ NaN` is false and `NaN < 0` is false in IEEE 754, so
  `signNoZero(NaN) = 0 - 0 = 0`. **This is a known divergence**: the Lean model returns
  1 for all non-negative inputs including NaN (which is not modelled), while the C++ would
  return 0 for NaN. Since the safety theorem `signNoZero_ne_zero` is a key property
  (ensures no division-by-zero in callers), maintainers should be aware that the C++
  version CAN return 0 for NaN float inputs.

**Impact on proofs**: The 6 theorems hold for the integer model. The NaN divergence is a
genuine concern for float callers — the safety property `signNoZero_ne_zero` does NOT
hold for C++ float inputs when `val` is NaN.

**Recommendation**: Callers that use `signNoZero<float>` and divide by the result should
ensure NaN cannot reach `signNoZero`, or the C++ should be patched to assert/clamp NaN.

### `PX4.Math.countSetBits`

| Lean name | C++ name | C++ location | Correspondence | Notes |
|-----------|----------|--------------|---------------|-------|
| `PX4.Math.countSetBits` | `math::countSetBits<T>` | [`src/lib/mathlib/math/Functions.hpp#L276`](../src/lib/mathlib/math/Functions.hpp#L276) | **abstraction** | Nat model; C++ uses while loop with bit shift |

**C++ source**:
```cpp
template<typename T>
int countSetBits(T n) {
    int count = 0;
    while (n) { count += n & 1; n >>= 1; }
    return count;
}
```

**Lean definition**: Structural recursion on `Nat` — `countSetBits 0 = 0`,
`countSetBits (n+1) = (n+1) % 2 + countSetBits ((n+1) / 2)`.

**Divergences**:
- **Type**: C++ template; Lean uses `Nat`. The C++ template is typically instantiated
  with `uint32_t` or `uint64_t`. For these types the semantics match exactly.
- **Signed types**: If the C++ template is instantiated with a signed type and `n < 0`,
  the right shift `n >>= 1` is implementation-defined in C++. The Lean model only
  applies to `Nat` (non-negative values).
- **Loop vs recursion**: The C++ while loop is equivalent to the structural recursion for
  non-negative inputs (the loop terminates and extracts the same bits).

**Impact on proofs**: The 9 concrete `native_decide` examples and the inductive
`countSetBits_pow2` theorem are valid for `Nat`. The correspondence to C++ `uint32_t` and
`uint64_t` is exact for non-negative values.

| Lean name | C++ name | C++ location | Correspondence | Notes |

**C++ source** (relevant lines 79–87):
```cpp
Type update(const Type new_value, const float deltatime)
{
    const Type dvalue_desired = new_value - _value;
    const Type dvalue_max = _slew_rate * deltatime;
    const Type dvalue = math::constrain(dvalue_desired, -dvalue_max, dvalue_max);
    _value += dvalue;
    return _value;
}
```

**Lean model**:
```lean
def slewUpdate (current target max_step : Int) : Int :=
  let d := target - current
  current + if d < -max_step then -max_step
            else if d > max_step then max_step
            else d
```

**Divergences**:

1. **Mutable state elided**: The C++ `update` method mutates `_value`. The Lean model
   takes `current` (= `_value` before the call) as an explicit argument and returns the
   new value. The correspondence is: `slewUpdate(current, target, max_step)` = value of
   `_value` after `update(target, dt)` where `max_step = _slew_rate * dt`.

2. **Float arithmetic**: C++ uses `float`/`double` for `Type` and `deltatime`. The Lean
   model uses `Int`. This abstracts away IEEE 754 rounding, NaN, and infinity. In
   practice the slew rate operates on actuator setpoints that are bounded floats, so NaN
   is not expected, but the proof does not cover it.

3. **`_slew_rate * deltatime` product**: The Lean model takes `max_step` as a single
   integer rather than computing the product. This abstracts the multiplication. For the
   theorems to apply to C++, the caller must ensure `_slew_rate * deltatime` does not
   overflow or produce a negative result.

4. **Negative `deltatime`**: If `deltatime < 0`, then `dvalue_max < 0` and
   `math::constrain(d, -dvalue_max, dvalue_max)` has inverted bounds — undefined
   behaviour in the C++ model. All Lean theorems require `0 ≤ max_step`.

**Impact on proofs**:

|---------|------------------------|-------|
| `slewUpdate_change_ge` | Valid for non-NaN, non-negative `dt` | Bounds hold for integer model |
| `slewUpdate_change_le` | Valid for non-NaN, non-negative `dt` | Primary rate-limiting invariant |
| `slewUpdate_reaches_target` | Valid | If distance ≤ max_step, target is reached exactly |
| `slewUpdate_nondecreasing` | Valid | When moving upward, no backwards steps |
| `slewUpdate_nonincreasing` | Valid | When moving downward, no backwards steps |
| `slewUpdate_no_overshoot_up` | Valid | No overshoot upward |

The primary safety invariant (no overshoot, rate bounded) is validly captured for the
integer model. Float rounding could cause sub-LSB differences but not directional

## Known Mismatches

**`signNoZero` and NaN** (severity: medium):

The C++ `math::signNoZero<float>` returns **0** for NaN inputs (both comparisons
`T(0) <= val` and `val < T(0)` are false for NaN, so the expression evaluates to
`0 - 0 = 0`). The Lean model does not model NaN and the `signNoZero_ne_zero` theorem
does not hold for C++ float NaN inputs.

**Affected theorem**: `signNoZero_ne_zero`

**Risk**: Any caller that uses `signNoZero<float>` result as a divisor or sign multiplier
without first checking for NaN could divide by zero if NaN propagates to the argument.
A search of `signNoZero` callers in the codebase is recommended.

**Recommended fix**: Add a `static_assert(!std::is_floating_point_v<T>)` to the C++
template (forcing callers to use integer types or to clamp NaN before calling), or
change the implementation to `return std::isfinite(val) ? ((T(0) <= val) ? 1 : -1) : 1`.
