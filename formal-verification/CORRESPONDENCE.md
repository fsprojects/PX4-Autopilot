| Lean name | C++ name | C++ location | Correspondence | Notes |
|-----------|----------|--------------|---------------|-------|
| `PX4.Math.signNoZero` | `math::signNoZero<T>` | [`src/lib/mathlib/math/Functions.hpp#L52`](../src/lib/mathlib/math/Functions.hpp#L52) | **abstraction** | Integer model; C++ uses bool subtraction |

**Divergences**:
- **Type**: C++ returns `int` from `bool` subtraction; Lean returns `Int`.
- **C++ expression**: `(T(0) <= val) - (val < T(0))` computes as `1 - 0 = 1` or

genuine concern for float callers — the safety property `signNoZero_ne_zero` does NOT
hold for C++ float inputs when `val` is NaN.

**Recommendation**: Callers that use `signNoZero<float>` and divide by the result should
ensure NaN cannot reach `signNoZero`, or the C++ should be patched to assert/clamp NaN.

### `PX4.Math.countSetBits`

| Lean name | C++ name | C++ location | Correspondence | Notes |
|-----------|----------|--------------|---------------|-------|
| `PX4.Math.countSetBits` | `math::countSetBits<T>` | [`src/lib/mathlib/math/Functions.hpp#L276`](../src/lib/mathlib/math/Functions.hpp#L276) | **abstraction** | Nat model; C++ uses while loop with bit shift |

**C++ source**:

```cpp
template<typename T>
int countSetBits(T n) {
    int count = 0;
    while (n) { count += n & 1; n >>= 1; }
    return count;
}
```

**Lean definition**: Structural recursion on `Nat` — `countSetBits 0 = 0`,
`countSetBits (n+1) = (n+1) % 2 + countSetBits ((n+1) / 2)`.

**Divergences**:

  the right shift `n >>= 1` is implementation-defined in C++. The Lean model only
  applies to `Nat` (non-negative values).
- **Loop vs recursion**: The C++ while loop is equivalent to the structural recursion for
  non-negative inputs (the loop terminates and extracts the same bits).

**Impact on proofs**: The 9 concrete `native_decide` examples and the inductive
`countSetBits_pow2` theorem are valid for `Nat`. The correspondence to C++ `uint32_t` and
`uint64_t` is exact for non-negative values.

| Lean name | C++ name | C++ location | Correspondence | Notes |

| `PX4.SlewRate.slewUpdate` | `SlewRate<Type>::update` | [`src/lib/slew_rate/SlewRate.hpp#L79`](../src/lib/slew_rate/SlewRate.hpp#L79) | **abstraction** | Pure functional model; state, float, and dt not modelled |

**C++ source** (relevant lines 79–87):
```cpp
Type update(const Type new_value, const float deltatime)
{
    const Type dvalue_desired = new_value - _value;
    const Type dvalue_max = _slew_rate * deltatime;
    const Type dvalue = math::constrain(dvalue_desired, -dvalue_max, dvalue_max);
    _value += dvalue;
    return _value;
}
```

**Lean model**:
```lean
def slewUpdate (current target max_step : Int) : Int :=
  let d := target - current
  current + if d < -max_step then -max_step
            else if d > max_step then max_step
            else d
```

**Divergences**:

1. **Mutable state elided**: The C++ `update` method mutates `_value`. The Lean model
   takes `current` (= `_value` before the call) as an explicit argument and returns the
   new value. The correspondence is: `slewUpdate(current, target, max_step)` = value of
   `_value` after `update(target, dt)` where `max_step = _slew_rate * dt`.

2. **Float arithmetic**: C++ uses `float`/`double` for `Type` and `deltatime`. The Lean
   model uses `Int`. This abstracts away IEEE 754 rounding, NaN, and infinity. In
   practice the slew rate operates on actuator setpoints that are bounded floats, so NaN
   is not expected, but the proof does not cover it.

3. **`_slew_rate * deltatime` product**: The Lean model takes `max_step` as a single
   integer rather than computing the product. This abstracts the multiplication. For the
   theorems to apply to C++, the caller must ensure `_slew_rate * deltatime` does not
   overflow or produce a negative result.

4. **Negative `deltatime`**: If `deltatime < 0`, then `dvalue_max < 0` and
   `math::constrain(d, -dvalue_max, dvalue_max)` has inverted bounds — undefined
   behaviour in the C++ model. All Lean theorems require `0 ≤ max_step`.

**Impact on proofs**:

| `slewUpdate_change_ge` | Valid for non-NaN, non-negative `dt` | Bounds hold for integer model |
| `slewUpdate_change_le` | Valid for non-NaN, non-negative `dt` | Primary rate-limiting invariant |
| `slewUpdate_reaches_target` | Valid | If distance ≤ max_step, target is reached exactly |
| `slewUpdate_nondecreasing` | Valid | When moving upward, no backwards steps |
| `slewUpdate_nonincreasing` | Valid | When moving downward, no backwards steps |
| `slewUpdate_no_overshoot_up` | Valid | No overshoot upward |
| `slewUpdate_no_overshoot_down` | Valid | No overshoot downward |

**C++ source** (condensed):
```cpp
const T deadzone(const T &value, const T &dz) {
    T x   = constrain(value, -1, 1);
    T dzc = constrain(dz, 0, 0.99);
    T out = (x - sign(x) * dzc) / (1 - dzc);

```

**Lean definition**:
```lean

  if x.abs > dz then (x - signR x * dz) / (1 - dz) else 0
```

**Divergences**:

1. **Input clamping elided**: The C++ clamps `value` to `[-1, 1]` and `dz` to `[0, 0.99]`

| `expo_at_zero` | Exact | `expo(0, e) = 0` always |

1. **Index computation fidelity**: The C++ truncates `value * (N-1)` to an integer via
   `(int)(...)`. For N=3 and rational `value`:
   - `value < 1/2` → `(int)(value * 2) = 0` → segment 0.
   - `value ≥ 1/2` → `(int)(value * 2) ≥ 1` → segment 1.
   The `if value < half` branch in `interpN3` captures this exactly for rationals.
   For floats, values very close to `1/2` could behave differently due to IEEE 754
   rounding of `value * 2` — this is the primary divergence for float instantiations.
2. **N=2 and N=3 only**: The C++ template is generic. The Lean model verifies only the
   N=2 and N=3 cases. For N > 3, the uniform-grid breakpoint structure (`k/(N-1)`) would
   require a more general inductive model.
3. **`constrain` in index computation**: The C++ calls `constrain((int)(value * (N-1)),
   0, N-2)` to clamp the index. For `value ∈ [0, 1]` with N=2, the index is always 0
   (no clamping needed). For N=3, the index is 0 or 1. The Lean `if value < half` branch
   subsumes the `constrain` clamp: values outside `[0, 1]` produce extrapolation from
   the appropriate end segment, matching C++ clamping behaviour.
4. **Type and NaN**: C++ is templated; Lean uses `Rat`. IEEE 754 NaN, infinity, and
   overflow are not modelled.

**Impact on proofs**: The 18 theorems cover both N=2 (5 theorems: endpoints, bounds,
range) and N=3 (13 theorems: node exactness, breakpoint continuity, range containment,
segment-wise monotonicity, constancy). The **breakpoint continuity** theorem
(`interpN3_continuity`) confirms no jump at `value = 1/2`. The **monotonicity** theorems
(`interpN3_mono_seg0`, `interpN3_mono_seg1`) confirm that the piecewise-linear function
preserves input order within each segment. All 18 theorems are sorry-free.

## Known Mismatches

**`negate<int16_t>` involution failure** (severity: low–medium):

The Lean 4 proof `negate16_not_involution` formally establishes that
`math::negate<int16_t>` is **not** a global involution:

```
negate(negate(-32767)) = negate(INT16_MAX) = INT16_MIN = -32768 ≠ -32767
```

Counterexample (bit pattern 32769 = -32767):
- `negate(32769)` = 32767 (= INT16_MAX), via the standard `return -value` branch
- `negate(32767)` = 32768 (= INT16_MIN), via the `if (value == INT16_MAX)` special case
- `negate(negate(32769))` = 32768 ≠ 32769

**Root cause**: The `INT16_MAX → INT16_MIN` special case is incorrect. In standard two's
complement, `-INT16_MAX = -32767` is representable in `int16_t` without overflow, so the
special case is unnecessary and causes round-trip failure for any value whose negation
equals INT16_MAX (i.e., for `value = -32767`).

**The only value needing a special case** is `INT16_MIN = -32768`, since
`-INT16_MIN = +32768` overflows `int16_t`.

**Correct fix**:
```cpp
template<>
constexpr int16_t negate<int16_t>(int16_t value)
{
    if (value == INT16_MIN) {
        return INT16_MAX;  // -(-32768) would overflow; saturate to INT16_MAX
    }
    return -value;
}
```

**Involution holds for**: all `int16_t` values except `value = -32767` under the
current (buggy) implementation. With the fix, it would hold universally.

**Risk in practice**: IMU drivers (`ICM42670P.cpp:608-634`, `BMI085_Accelerometer.cpp:504-505`)
use `math::negate` to flip sensor axes. A value of exactly -32767 (= 0x8001) in the
raw sensor data would be incorrectly round-tripped if negation is applied twice. This
is unlikely in practice (raw IMU data must hit exactly -32767) but represents a
correctness defect.

**Related theorem**: `negate16_not_involution`, `negate16_counterexample` in
`formal-verification/lean/FVSquad/Negate.lean`.

**`signNoZero` and NaN** (severity: medium):

The C++ `math::signNoZero<float>` returns **0** for NaN inputs (both comparisons
`T(0) <= val` and `val < T(0)` are false for NaN, so the expression evaluates to
`0 - 0 = 0`). The Lean model does not model NaN and the `signNoZero_ne_zero` theorem
does not hold for C++ float NaN inputs.

**Affected theorem**: `signNoZero_ne_zero`

**Risk**: Any caller that uses `signNoZero<float>` result as a divisor or sign multiplier
without first checking for NaN could divide by zero if NaN propagates to the argument.
A search of `signNoZero` callers in the codebase is recommended.

**Recommended fix**: Add a `static_assert(!std::is_floating_point_v<T>)` to the C++
template (forcing callers to use integer types or to clamp NaN before calling), or
change the implementation to `return std::isfinite(val) ? ((T(0) <= val) ? 1 : -1) : 1`.

**Lean model**:

```lean

  requested : Bool
  delayTF   : Nat   -- delay true→false (µs)
  delayFT   : Nat   -- delay false→true (µs)
```

The C++ class additionally carries `_last_time_to_change_state : hrt_abstime`, which
marks when the last state-change request arrived. In the Lean model this is captured
implicitly through the `now` argument to `hysteresisUpdate` and the inequality

instead, theorems are stated with explicit timing hypotheses (e.g.,
`now ≥ h.lastChange + h.delayTF` for a true→false commit). This is a deliberate
abstraction: it lets us prove timing properties without encoding the full timestamp-update
logic, at the cost of making initial-change-time a universally-quantified hypothesis.

1. **Timestamp type**: C++ uses `hrt_abstime` (64-bit µs counter). Lean uses `Nat`

2. **Mutable state**: C++ `Hysteresis` is a mutable class. The Lean model is a pure

   preconditions are stated explicitly in each theorem.
3. **Thread safety**: The C++ implementation is not thread-safe. The Lean model has no
   concurrency; theorems apply to single-threaded update sequences.
4. **`_last_time_to_change_state` update**: In C++, `_last_time_to_change_state` is

   `update_settled_noop` covers the `requested = state` case but the timing-reset
   behaviour on re-request is not directly modelled.
5. **Clock monotonicity**: C++ assumes `now_us` is non-decreasing. The Lean model makes
   no such assumption; theorems hold for any `now : Nat`.

hold within the abstraction. The elision of `lastChange` tracking means the model
cannot directly prove a bounded-response-time property for a sequence of updates, but
this is a known gap, not a mismatch.

| Theorem | Property |

```c
uint64_t crc64_add_word(uint64_t crc, uint32_t value)
{
    uint32_t i, j;
    uint8_t byte;
    const uint64_t poly = 0x42F0E1EBA9EA3693ull;

    for (j = 0; j < 4; j++) {
        byte = ((uint8_t *) &value)[j];    // little-endian: j=0 is LSB
        crc ^= (uint64_t) byte << 56u;     // byte into MSB

        for (i = 0; i < 8; i++) {
            if (crc & (1ull << 63u)) {
                crc = (uint64_t)(crc << 1u) ^ poly;
            } else {
                crc = (uint64_t)(crc << 1u);
            }
        }
    }
    return crc;
}
```
