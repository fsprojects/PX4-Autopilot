# Formal Verification Research

🔬 *Generated by Lean Squad automated formal verification.*

## Overview

This document surveys PX4-Autopilot for formal verification targets using Lean 4.

PX4-Autopilot is a flight control stack written primarily in C++ targeting embedded
systems. It includes a rich `mathlib` library of pure mathematical utilities used
throughout the flight controller. These utility functions are ideal FV targets because:

- They are **pure** (no side effects, no I/O)
- They have **clear specifications** (bounded outputs, monotonicity, etc.)
- They are **safety-critical** (used in control loops, sensor fusion, attitude estimation)
- Bugs in them would propagate throughout the system silently

## Tool Choice: Lean 4

We use **Lean 4** with its standard library. Mathlib is not available via the network
firewall in this environment, so we use only `lean4` standard tactics (`omega`, `simp`,
`cases`, `induction`, `decide`, `native_decide`).

This is sufficient for the identified targets, which involve:
- Integer arithmetic and bit manipulation
- Ordered comparison and clamping
- Linear arithmetic properties

## Identified Targets

### 1. `math::constrain<T>` (Priority: HIGH)

**File**: `src/lib/mathlib/math/Limits.hpp`

```cpp
template<typename _Tp>
constexpr _Tp constrain(_Tp val, _Tp min_val, _Tp max_val)
{
    return (val < min_val) ? min_val : ((val > max_val) ? max_val : val);
}
```

**Benefit**: This function is called thousands of times throughout PX4, including in
safety-critical paths like the rate controller, position controller, and sensor fusion.
Verifying its contract (idempotence, monotonicity, output-in-range) gives broad assurance.

**Key properties to verify**:
- Output is always in `[min_val, max_val]` (when `min_val ≤ max_val`)
- Idempotence: `constrain(constrain(v, lo, hi), lo, hi) = constrain(v, lo, hi)`
- Monotonicity: if `v1 ≤ v2` then `constrain(v1, lo, hi) ≤ constrain(v2, lo, hi)`
- Fixed point: if `lo ≤ v ≤ hi` then `constrain(v, lo, hi) = v`

**Specification size**: ~20 Lean lines for properties, ~5 for implementation model
**Proof tractability**: High — `omega` handles all Int cases; `Int.le_max` etc.
**Approximations**: We model with `Int` instead of `float` to avoid floating-point
complications. Float version requires separate treatment of NaN.

### 2. `math::signNoZero<T>` (Priority: HIGH)

**File**: `src/lib/mathlib/math/Functions.hpp`

```cpp
template<typename T>
int signNoZero(T val)
{
    return (T(0) <= val) - (val < T(0));
}
```

**Benefit**: A sign function that returns `{-1, 1}` (never zero). Used in control loops
where division by sign is performed. If it ever returned 0 due to a bug, it would cause
division by zero or incorrect control. Verifying it never returns 0 is safety-relevant.

**Key properties to verify**:
- Output is always 1 or -1 (never 0)
- Correct sign: `signNoZero(v) = 1 ↔ 0 ≤ v`
- Correct sign: `signNoZero(v) = -1 ↔ v < 0`

**Specification size**: ~15 Lean lines
**Proof tractability**: High — `omega` closes all goals on Int
**Approximations**: Int model; float version requires reasoning about total order

### 3. `math::countSetBits<T>` / Hamming weight (Priority: MEDIUM)

**File**: `src/lib/mathlib/math/Functions.hpp`

```cpp
template<typename T>
int countSetBits(T n)
{
    int count = 0;
    while (n) { count += n & 1; n >>= 1; }
    return count;
}
```

**Benefit**: Used for bit manipulation in hardware configuration and CAN bus management.
The correctness of Hamming weight is a textbook property, but the loop-based
implementation requires an inductive argument.

**Key properties to verify**:
- Result is always non-negative
- Result is bounded by bit width
- Concrete examples (spot-check correctness via `native_decide`)
- The power-of-2 characterisation: `countSetBits(2^k) = 1`

**Specification size**: ~30 Lean lines
**Proof tractability**: Medium — inductive argument on bit length needed for pow2 case
**Approximations**: We model with `Nat` and use a recursive functional definition

### 4. `SlewRate::update` (Priority: MEDIUM)

**File**: `src/lib/slew_rate/SlewRate.hpp`

**Key idea**: The slew rate limiter clamps the per-step change in a value to a maximum
rate. It is used throughout flight controllers to prevent actuator damage.

**Key properties to verify**:
- The change per step is bounded: `|update(v, dt) - state| ≤ slew_rate * dt`
- If the desired value is reachable in one step, the output equals the desired value
- Monotone tracking: the output converges toward the target

**Specification size**: ~25 Lean lines
**Proof tractability**: Medium — depends on `constrain` properties
**Approximations**: Pure functional model; time is a parameter, no accumulation

---

### 5. `math::interpolate` (Priority: LOW-MEDIUM)

**File**: `src/lib/mathlib/math/Functions.hpp`

**Key properties**: Output is in `[y_low, y_high]` when input is in `[x_low, x_high]`; 
monotone; boundary conditions exact.

**Proof tractability**: Medium-hard for floats; easier with rationals, but requires
numeric abstraction.

## Approach Notes

1. **Integer model**: We use `Int` or `Nat` to model floating-point operations. This
   captures the logical structure without requiring IEEE 754 reasoning.
2. **Standard library only**: No Mathlib dependency — avoids network issues and speeds
   up builds. Standard `omega` and `simp` tactics are sufficient.
3. **Decidable properties**: Where possible, we formulate properties so `native_decide`
   can check them on concrete inputs as regression tests.
4. **Incremental proofs**: We use `sorry` for hard sub-goals and prove what we can,
   then iterate in subsequent runs.
