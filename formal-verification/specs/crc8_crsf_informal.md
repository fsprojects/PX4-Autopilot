## Purpose

Computes a CRC-8 checksum over a byte buffer using the CRSF (Crossfire RC)
protocol polynomial **0xD5** (MSB-first, non-reflected). The checksum is appended
to or compared against CRSF telemetry and RC packets to detect data corruption
over the serial link.

The implementation is **table-driven**: `Crc8Init` precomputes a 256-entry LUT
from the polynomial, and `Crc8Calc` processes each byte with a single XOR + table
lookup. This is equivalent to — but faster than — re-running the 8-bit shift loop
for every input byte.

## Preconditions

1. `Crc8Init(0xD5)` has been called exactly once before `Crc8Calc` is first used.
   (In the Lean model we eliminate global state by computing the LUT entry inline.)
2. `data` is a valid pointer to a contiguous array of at least `size` bytes.
   (Abstracted in Lean as `List UInt8`.)
3. No preconditions on the content of the buffer.

## Postconditions

1. **Return type**: the result is a `uint8_t` in `[0, 255]`.
2. **Empty buffer**: `Crc8Calc([], 0) = 0` — the initial CRC state is 0.
3. **Deterministic**: the same byte sequence always yields the same CRC value.
4. **Fold/split (incremental computation)**:
   For any two buffers `a` and `b`:
   ```
   Crc8Calc(a ++ b) = crc8Continue(Crc8Calc(a), b)
   ```
   where `crc8Continue(init, buf)` is the same algorithm with a non-zero start.
5. **Single byte**: `Crc8Calc([b]) = crc8Step(0, b)` for any byte `b`.

## Invariants

- The computation is equivalent to `List.foldl crc8Step 0 bytes` for any input list.
- The fold/split property makes incremental computation correct: checksumming a
  packet in pieces gives the same result as checksumming it all at once.

## Algorithm Details

### Table initialisation (`Crc8Init`)

For each `idx ∈ [0, 255]`:
```cpp
uint8_t crc = idx;
for (int shift = 0; shift < 8; ++shift) {
    crc = (crc << 1) ^ ((crc & 0x80) ? poly : 0);
}
crc8_lut[idx] = crc;
```
This is the standard bit-at-a-time CRC computation for polynomial `poly` (0xD5),
MSB-first. The shift loop applies the polynomial one bit at a time.

### CRC computation (`Crc8Calc`)

```cpp
uint8_t crc = 0;
while (size--) {
    crc = crc8_lut[crc ^ *data++];
}
return crc;
```

Per-byte update: `crc_new = crc8_lut[crc ^ b]`

## Lean 4 Modelling Approach

The two-layer structure (init + calc) is collapsed into a single pure function
by inlining the LUT entry computation:

1. **Polynomial step** (`crc8PolyStep`): one iteration of the shift loop:
   ```lean
   def crc8PolyStep (crc : UInt8) : UInt8 :=
     (crc <<< 1) ^^^ (if crc &&& 0x80 != 0 then 0xD5 else 0)
   ```

2. **LUT entry** (`crc8LutEntry`): apply `crc8PolyStep` 8 times to the index:
   ```lean
   def crc8LutEntry (idx : UInt8) : UInt8 :=
     (List.range 8).foldl (fun crc _ => crc8PolyStep crc) idx
   ```

3. **Per-byte step** (`crc8Step`):
   ```lean
   def crc8Step (crc : UInt8) (b : UInt8) : UInt8 :=
     crc8LutEntry (crc ^^^ b)
   ```

4. **Buffer CRC** (`crc8`):
   ```lean
   def crc8 (bs : List UInt8) : UInt8 := bs.foldl crc8Step 0
   ```

5. **Key theorem** (fold/split):
   ```
   theorem crc8_append (a b : List UInt8) :
       crc8 (a ++ b) = crc8Continue (crc8 a) b
   ```
   This follows directly from `List.foldl_append`.

## Edge Cases

| Scenario | Expected result |
|----------|----------------|
| Empty buffer | `0` |
| Single zero byte | `0` (no bits set, polynomial XOR does nothing) |
| Any `[b]` | `crc8Step 0 b = crc8LutEntry b` |
| Split at any position | same result as full computation |

## Examples

- `crc8 [] = 0`
- `crc8 [b] = crc8Step 0 b` for any `b : UInt8`
- `crc8 (a ++ b) = crc8Continue (crc8 a) b`
- `crc8 [0x00] = 0`
- Concrete values can be verified with `native_decide`

## Inferred Intent

The CRSF protocol uses this CRC-8 to detect transmission errors on RC link
telemetry and control packets. The 0xD5 polynomial (x⁸ + x⁷ + x⁶ + x⁴ + x² + 1)
is the CRC-8/DVB-S2 polynomial, known for good error-detection on short packets
typical in RC control systems.

The fold/split property is the central formal guarantee: it ensures that the
table-driven LUT implementation computes the same value as the naive bit-at-a-time
algorithm, and that incremental computation over packet chunks is correct.
