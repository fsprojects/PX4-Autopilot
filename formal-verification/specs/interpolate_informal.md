## Purpose

`interpolate` is a **clamped linear interpolation** function. Given two key-value pairs
`(x_low, y_low)` and `(x_high, y_high)`, it maps an input `value` to an output on the
piecewise-linear curve:

- For inputs **below or at** `x_low`: output is clamped to `y_low`.
- For inputs **above** `x_high`: output is clamped to `y_high`.
- For inputs **strictly between** `x_low` and `x_high` (inclusive of `x_high`): output
  is the linear interpolation between `y_low` and `y_high`, scaled by the fraction of
  how far `value` is through the `[x_low, x_high]` interval.

Used extensively throughout PX4 for sensor scaling, control curve mapping, and parameter
interpolation (e.g., throttle curves, battery discharge curves, actuator mixing).

- **`value = x_low`**: Returns `y_low` via the `â‰¤` branch. Note the asymmetry: the lower
  boundary is inclusive (`â‰¤`), while the upper boundary is exclusive (`>`). So `x_high`
  itself falls through to the linear formula (postcondition 4 proves this equals `y_high`).

- **`y_low > y_high`**: The function still works; the slope is negative. Monotonicity
  becomes non-increasing. The output range is `[y_high, y_low]`.

## Open Questions

1. **Asymmetric boundary**: Is the inclusive-low / exclusive-high convention intentional?
   For most callers, either convention at the exact boundary should produce the same
   result (since both branches yield y_low or y_high respectively when
   `value = x_low = x_high` would be degenerate anyway).
2. **Inverted ranges**: The spec does not currently handle `x_low > x_high`
   (undefined/unspecified in C++). Should the function defensively saturate, or is the
   caller required to ensure `x_low < x_high`?
