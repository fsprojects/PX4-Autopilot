## Purpose

A sliding-window **median filter** that rejects outliers by returning the median
(middle sorted value) of the most recent `WINDOW` samples. Used in PX4 for sensor
noise filtering where occasional spikes need to be rejected without distorting the
running signal. The window size is a compile-time template parameter that must be
odd (‚Č• 3).

---

## Preconditions

- `WINDOW` is a positive odd integer ‚Č• 3 (enforced by `static_assert`).
- For floating-point `T`, values that are not finite (`NaN`, `¬Ī‚ąě`) are handled by
  the custom `cmp` comparator: non-finite values are sorted to the top (larger end).
  This is not the standard IEEE 754 ordering. **Open question**: is this the intended
  behaviour for NaN inputs? A NaN in the window will propagate to the median output if
  `WINDOW/2` or more elements are NaN.
- The buffer is zero-initialised (`_buffer[WINDOW] {}`). Before `WINDOW` samples have
  been inserted, the buffer contains zeros in the un-inserted slots.

## Postconditions

### `insert(sample)`
- **Buffer update**: `_head` advances by 1 modulo `WINDOW`, and `sample` replaces the
  oldest entry at the new `_head` position.
- **Circular write**: the circular index `_head` always satisfies `0 ‚Č§ _head < WINDOW`.
- **Side effect only**: no return value.

### `median()`
- **Non-destructive**: does not modify `_buffer` or `_head`.
- **Returns sorted[WINDOW/2]**: creates a local copy, sorts it, and returns the element
  at index `WINDOW/2` (integer division). For `WINDOW = 3` this is index 1 (the middle
  element); for `WINDOW = 5` this is index 2.
- **Bounded**: the returned value is always ‚Č• the minimum of the current buffer and ‚Č§
  the maximum of the current buffer.
- **Sorted-position**: the returned value `m` satisfies:
  - At least `(WINDOW+1)/2` elements of the buffer are ‚Č§ `m`.
  - At least `(WINDOW+1)/2` elements of the buffer are ‚Č• `m`.
- **Constant window**: median([x, x, ‚Ä¶, x]) = x for any x.

### `apply(sample)`
- Equivalent to `insert(sample)` followed by `median()`.
- **Spike rejection**: if all buffer slots except one contain the same value `v`, and
  one slot contains an outlier `o ‚Č† v`, then `apply` returns `v` (not `o`). This holds
  for `WINDOW = 3` (2/3 elements dominate) and more generally whenever `WINDOW/2 + 1`
  elements share the same value.

## Invariants

- `_head` always satisfies `0 ‚Č§ _head < WINDOW` (preserved by `(_head + 1) % WINDOW`).
- The buffer contains exactly the last `WINDOW` inserted samples (or initial zeros for
  positions not yet written).
- The circular structure is FIFO: `insert` always overwrites the slot that has not been
  written to for the longest time.

## Edge Cases

| Scenario | Expected behaviour |
|----------|--------------------|
| First call to `median()` (no inserts yet) | Returns 0 (all buffer slots are zero-initialised) |
| Window = 3, two duplicate values | Returns the duplicate (outlier rejected) |
| All window slots contain the same value `v` | Returns `v` |
| Window = 3, all distinct values `a < b < c` | Returns `b` (the middle value) |
| Floating-point `T`, one NaN in window (WINDOW=3) | The NaN sorts to top; non-NaN middle element returned if at least 2 are finite |
| Two or more NaN values in a 3-element window | NaN returned (2/3 NaN ‚Üí sorted middle = NaN) |
| `WINDOW/2` = 1 for WINDOW=3 | Returns `sorted[1]` ‚ÄĒ the middle element |

## Concrete Examples

For `WINDOW = 3`, buffer = `[1, 5, 2]`:
- `sorted = [1, 2, 5]`, `sorted[1] = 2` ‚Üí `median()` returns `2`.

For `WINDOW = 3`, buffer = `[10, 2, 3]` (spike = 10):
- `sorted = [2, 3, 10]`, `sorted[1] = 3` ‚Üí spike 10 is rejected.

For `WINDOW = 5`, buffer = `[3, 1, 4, 1, 5]`:
- `sorted = [1, 1, 3, 4, 5]`, `sorted[2] = 3` ‚Üí `median()` returns `3`.

## Inferred Intent

The filter is designed for **spike suppression** in sensor pipelines (e.g., range
finders, baro altitude). A single corrupted reading does not change the output for
`WINDOW = 3` unless two consecutive corrupted readings arrive. The implementation
copies the buffer before sorting (preserving insertion history) and uses C `qsort`
with a custom comparator. The comparator places non-finite values after all finite
values, treating NaN as "very large".

---

## Open Questions

1. **NaN propagation**: is returning a non-NaN value when one of three elements is NaN
   the intended behaviour? The custom `cmp` sorts NaN to the top, so with one NaN in
   a 3-element window, the median is always finite. This could mask a sensor failure.
2. **Zero-initialisation bias**: the initial zeros can distort early readings if the
   signal is far from zero. Should callers pre-fill the buffer (e.g., via `WINDOW`
   initial `insert` calls) before trusting `median()`? There is no explicit API for
   this.
3. **Float vs integer**: the `cmp` function treats `<` as the total order for integers
   and uses `__builtin_isfinite` for floats. For integers this is always a total order.
   For floats the NaN treatment is non-standard. Is this intentional?
