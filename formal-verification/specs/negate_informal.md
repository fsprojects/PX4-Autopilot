## Purpose

`math::negate<int16_t>` negates an `int16_t` value without causing signed integer
overflow. In standard C++, negating `INT16_MIN = -32768` is undefined behaviour (the
result `+32768` is not representable in `int16_t`). The specialisation handles this
boundary case safely.

The function is used by IMU sensor drivers to flip sensor axes (e.g., negate the Y and
Z accelerometer/gyroscope readings when the sensor is mounted with an inverted axis).

```cpp
template<>
constexpr int16_t negate<int16_t>(int16_t value)
{
    if (value == INT16_MAX) {
        return INT16_MIN;
    } else if (value == INT16_MIN) {
        return INT16_MAX;
    }
    return -value;
}
```

- `value` is any `int16_t` (no restrictions — the function is total).

## Inferred Intent

The intended semantics is **overflow-safe negation**: avoid undefined behaviour on
`INT16_MIN`. The `INT16_MIN → INT16_MAX` special case is correct for this purpose.

The `INT16_MAX → INT16_MIN` special case appears to be a mistake. `-INT16_MAX = -32767`
is representable in `int16_t` without overflow, so no special case is needed. The
likely intent was to make the function a bijection (paired extremes), but this choice
creates an incorrect value for `INT16_MAX` inputs.

## Bug Found via Formal Verification

The Lean 4 proof `negate16_not_involution` shows the function is **not a global
involution**:

```
negate(negate(-32767)) = negate(INT16_MAX) = INT16_MIN = -32768 ≠ -32767
```

This is counterexample `value = -32767` (bit pattern `0x8001`).

**Correct implementation**:
```cpp
template<>
constexpr int16_t negate<int16_t>(int16_t value)
{
    if (value == INT16_MIN) {  // Only this case needs a special-case
        return INT16_MAX;
    }
    return -value;
}
```
