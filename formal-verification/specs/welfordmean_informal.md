## Purpose

`WelfordMean<T>` implements Welford's online algorithm for computing the running
mean and variance of a stream of values without storing all samples. After `n`
calls to `update(xᵢ)`, the object holds:

- `_count` = `n`
- `_mean` ≈ arithmetic mean of `x₁, ..., xₙ`
- `_M2` ≈ sum of squared deviations `Σᵢ(xᵢ − mean)²`

The primary value is memory efficiency: only three values are stored regardless
of how many samples have been seen. The algorithm is numerically more stable than
naïve summation for large `n` or values far from zero.

## Class State

| Field | Type | Initial value | Meaning |
|-------|------|---------------|---------|
| `_count` | `uint16_t` | 0 | number of samples seen |
| `_mean` | `Type` | 0 | running mean |
| `_M2` | `Type` | 0 | running sum of squared deviations from mean |
| `_mean_accum` | `Type` | 0 | Kahan compensator for `_mean` |
| `_M2_accum` | `Type` | 0 | Kahan compensator for `_M2` |

After `reset()`, all fields return to their initial values.

---

## Preconditions

- `new_value` must be finite (non-NaN, non-infinity) for the invariants to hold.
  The implementation checks `PX4_ISFINITE` and calls `reset()` if either `_mean`
  or `_M2` becomes non-finite after an update.
- The algorithm is defined for all real-valued inputs; integer overflow in
  `_count` is handled by a special case at `UINT16_MAX`.

---

## Postconditions (mathematical model, ignoring floating-point rounding)

Let `{x₁, x₂, ..., xₙ}` be the values passed to `update` in order.
After `n` calls, the ideal state satisfies:

1. **Count**: `_count = n`

2. **Mean invariant**: `_mean = (x₁ + x₂ + ... + xₙ) / n`

3. **M2 invariant**: `_M2 = Σᵢ₌₁ⁿ (xᵢ − mean)²`
   (sum of squared deviations from the current mean)

4. **Recurrence (mean)**: for `n ≥ 2`,
   ```
   meanₙ = meanₙ₋₁ + (xₙ − meanₙ₋₁) / n
   ```
   This is the defining recurrence of Welford's algorithm.

5. **Recurrence (M2)**: for `n ≥ 2`,
   ```
   M2ₙ = M2ₙ₋₁ + (xₙ − meanₙ₋₁) · (xₙ − meanₙ)
   ```
   Note: two different means appear — the mean before and after update.

6. **M2 non-negativity**: `_M2 ≥ 0` (enforced by `math::max(_M2, 0.f)` in
   the implementation; the mathematical invariant also holds since `M2` is a
   sum of squares).

7. **Variance**: `variance() = _M2 / (_count − 1)` (sample variance, Bessel-corrected).
   Valid only when `_count ≥ 2`; the implementation requires `_count > 2` for `valid()`.

---

## Invariants

- **Non-negativity**: `_M2 ≥ 0` always (both enforced by `max` and mathematically true).
- **Mean bounds**: for bounded input `|xᵢ| ≤ B`, the mean satisfies `|_mean| ≤ B`.
- **Welford recurrence**: at every step, `M2ₙ = M2ₙ₋₁ + δ_old · (xₙ − meanₙ)`
  where `δ_old = xₙ − meanₙ₋₁`.

---

## Edge Cases

| Case | Behaviour |
|------|-----------|
| `_count = 0` (first call) | `reset()` + `_count = 1`, `_mean = new_value`, returns `false` |
| `_count = UINT16_MAX` (overflow) | `_M2 /= _count`, `_mean_accum = 0`, `_count = 1` (mean preserved, variance scale-reset) |
| NaN or inf in `_mean` or `_M2` | `reset()`, returns `false` |
| `_count = 1` | `variance()` would divide by 0; `valid()` returns false until `_count > 2` |
| `_count = 2` | `variance()` would divide by 1; `valid()` still returns false |
| All same values | `_M2 = 0`, `variance() = 0`, `_mean = x` |
| Two values `a`, `b` | `mean = (a+b)/2`, `M2 = (a−b)²/2` |

## Examples

### Example 1: Three values 1, 2, 3

After `update(1)`: `count=1`, `mean=1`, `M2=0`, `valid()=false`  
After `update(2)`: `count=2`, `mean=1.5`, `M2=0.5`, `valid()=false`  
After `update(3)`: `count=3`, `mean=2`, `M2=2`, `valid()=true`  
`variance() = 2 / 2 = 1` ✓ (sample variance of {1,2,3} = 1)

Recurrence check for third step:
- `delta_old = 3 − 1.5 = 1.5`
- `new_mean = 1.5 + 1.5/3 = 2`
- `delta_new = 3 − 2 = 1`
- `M2_new = 0.5 + 1.5 * 1 = 2` ✓

### Example 2: All same value k

After n calls: `mean = k`, `M2 = 0`, `variance() = 0`.

## Inferred Intent

The algorithm is intended to:
1. **Minimise numerical error** compared to `(Σxᵢ)²/n − Σxᵢ²` form
2. **Handle streaming data** with bounded memory
3. **Guard against overflow/NaN propagation** — the `PX4_ISFINITE` check and
   `max(_M2, 0)` clamp are defensive measures to keep the accumulator clean
4. **Enable calibration-time use** — called from IMU calibration routines where
   samples arrive one by one and mean/variance are needed at the end

---

## Open Questions

1. **Kahan summation modelling**: The Kahan compensator (`_mean_accum`, `_M2_accum`)
   reduces floating-point error but complicates the Lean model. A natural approach is
   to prove the abstract recurrence for `Rat` (ignoring Kahan), then separately note
   that the Kahan term only reduces rounding error and cannot affect the abstract
   correctness. **Question for maintainer**: is the Kahan sum essential to any known
   use case, or is it a belt-and-suspenders for noisy IMU data?

2. **`_count = UINT16_MAX` reset semantics**: When the count overflows, `_M2 /= _count`
   scales the M2 accumulator and resets count to 1. The mean is preserved. This breaks
   the standard Welford invariant: the new `_M2` does not correspond to the squared
   deviation from the preserved mean for 1 sample. The intent appears to be: "forget
   the detailed history but keep a good estimate of variance". Is this semantics
   intentional and documented? The Lean spec should model the non-overflow case and note
   this as an approximation.

3. **`valid()` threshold**: `valid()` requires `_count > 2`, not `_count ≥ 2`. This means
   3 samples are needed before results are considered valid. Is this a deliberate
   conservatism for numerical stability, or is `_count > 1` sufficient for the variance
   to be meaningful?

## Lean 4 Modelling Plan

**Approach**: Model the pure recurrence over `Rat`, ignoring Kahan (which is a numerical
stability detail), ignoring the `UINT16_MAX` overflow case, and assuming finite inputs.

**Key Lean definitions** to write:
```lean
-- State of the Welford accumulator after n updates
structure WelfordState where
  count : Nat
  mean : Rat
  M2 : Rat

-- Single-step update (pure recurrence)
def welfordUpdate (s : WelfordState) (x : Rat) : WelfordState :=
  let n := s.count + 1
  let delta_old := x - s.mean
  let new_mean := s.mean + delta_old / n
  let delta_new := x - new_mean
  { count := n, mean := new_mean, M2 := s.M2 + delta_old * delta_new }

-- Iterate n updates over a list of values
def welfordFold : List Rat → WelfordState :=
  List.foldl welfordUpdate { count := 0, mean := 0, M2 := 0 }
```

**Key theorems** to prove:
1. `welfordUpdate_count`: `(welfordUpdate s x).count = s.count + 1`
2. `welfordUpdate_mean`: `(welfordUpdate s x).mean = (s.mean * s.count + x) / (s.count + 1)`
   (i.e., the running mean equals the arithmetic mean of all seen values)
3. `welfordFold_mean`: `(welfordFold xs).mean = xs.sum / xs.length` (for `xs.length ≥ 1`)
4. `welfordUpdate_M2_nonneg`: `0 ≤ s.M2 → 0 ≤ (welfordUpdate s x).M2`
   (non-negativity is preserved — no need for the `max` clamp in the abstract model)
5. `welfordFold_M2_eq`: `(welfordFold xs).M2 = Σᵢ (xᵢ - mean)²` for the correct mean

**Tractability**: Theorems 1–4 should follow by `simp` + `omega`/`ring` on `Rat`. 
Theorem 5 requires a careful inductive argument expanding the summation — this is the
core mathematical content. A helper lemma expressing the shift in mean between steps
`n-1` and `n` would make the induction cleaner.

**Mathlib needed?**: Not for theorems 1–4. Theorem 5 may benefit from `Mathlib.Algebra.BigOperators`
for the sum notation, but can also be stated for a `List` sum using stdlib folds.
